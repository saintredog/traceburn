## 1. Language & Stack Decision

### Recommendation: Python 3.11+

Python 3.11+ is the recommended implementation language for TraceBurn.

### Rationale vs. Alternatives

| Language | Verdict | Reason |
|----------|---------|--------|
| **Python 3.11+** | ✅ Selected | Mature browser automation (Playwright), rich CLI ecosystem (Click, Rich), best-in-class scraping/HTTP libraries, rapid iteration, wide contributor base, strong cryptography support |
| Node.js | ❌ Rejected | Playwright is also available in Node, but async callback complexity increases maintenance burden; fewer data-science-adjacent libraries; packaging (npm) is harder to bundle into a self-contained installer |
| Go | ❌ Rejected | Excellent binary distribution, but browser automation (Rod/Chromedp) is significantly less mature than Python's Playwright; form-filling workflows require more low-level work; smaller contributor pool for this domain |
| Rust | ❌ Rejected | No production-ready browser automation; development speed too slow for a tool that must iterate rapidly on broker-specific workflows; overkill for I/O-bound scraping tasks |

Python's Playwright library is the most actively maintained, best-documented, and most widely used browser automation library across any language. For a tool whose core value is browser-based form submission and scraping, this is a decisive advantage.

### Core Dependencies

| Package | Version | Purpose |
|---------|---------|---------|
| `playwright` | `>=1.43` | Browser automation — Chromium/Firefox headless; used for broker form submission (Tier 1 removal) and screenshot/proof capture |
| `requests` | `>=2.31` | HTTP client for lightweight API calls, broker health checks, and email opt-out endpoints (Tier 2) |
| `click` | `>=8.1` | CLI framework — command routing, argument parsing, help text generation |
| `cryptography` | `>=42.0` | AES-256-GCM encryption for the PII vault; Fernet key derivation via PBKDF2HMAC |
| `sqlite3` | built-in | Embedded relational database — no server required; stores exposures, brokers, removal requests, proof records |
| `rich` | `>=13.7` | Terminal UI — progress bars, tables, colored status output, markdown rendering |
| `pydantic` | `>=2.6` | Data model validation — broker definitions, exposure records, removal request schemas |
| `apscheduler` | `>=3.10` | Task scheduling — post-removal re-verification at 7/30/90-day intervals; background re-scan jobs |
| `pyyaml` | `>=6.0` | Parse `brokers.yaml` and `user.yaml` config files |
| `jinja2` | `>=3.1` | Email template rendering for Tier 2 opt-out emails |
| `python-dotenv` | `>=1.0` | Environment variable loading for local config |

## 2. CLI vs. Web Dashboard vs. Hybrid

### Recommendation: CLI-First; Web Dashboard as Phase 2 Roadmap Item

### Rationale

TraceBurn is a **privacy tool**. Its architecture must not contradict its purpose:

- A web dashboard requires a server process, which means open ports, authentication surfaces, and the temptation (or vendor pressure) to move storage to the cloud.
- A subscription SaaS model (like competitors) forces the user to trust a third party with their most sensitive PII. TraceBurn refuses this trade-off.
- CLI-first means the tool runs entirely on the user's machine, uses no network for its own operation (only to contact data brokers), and leaves no external footprint.

### Local-First Philosophy

1. **All data stays on the user's filesystem** — encrypted SQLite, user-defined location.
2. **No account, no API key, no telemetry** — TraceBurn phones home to no one.
3. **Air-gap compatible** — advanced users can run TraceBurn on a machine with no inbound connections.
4. **User owns the export** — encrypted backup is a file the user controls.

### Phase 2 Roadmap: Optional Local Dashboard

A future optional web dashboard will be strictly `localhost`-only:

- Framework: **FastAPI** (async, fast startup, automatic OpenAPI docs for power users)
- Launch: `traceburn dashboard` — starts a local server, opens browser to `http://127.0.0.1:7734`
- No external binding, no auth tokens stored remotely, CORS locked to localhost
- Dashboard is read-only by default; write actions still require CLI confirmation

## 3. Data Storage Strategy

### Recommendation: SQLite with Application-Layer AES-256-GCM Encryption

### Why SQLite

- Zero-configuration, zero-server, single-file database
- Built into Python's standard library (`sqlite3`)
- Sufficient performance for thousands of broker records and hundreds of removal requests
- Easy to backup (copy one file), easy to inspect (DB Browser for SQLite), easy to delete

### Encryption Approach

Application-layer encryption using the `cryptography` library (AES-256-GCM):

- The database file itself is stored encrypted at rest using a key derived from the user's master passphrase via PBKDF2HMAC (SHA-256, 600,000 iterations)
- The vault key is never stored to disk — derived at runtime from passphrase entry
- Alternative: SQLCipher for full transparent block-level encryption (evaluated but adds a C extension dependency that complicates packaging; application-layer is simpler and auditable)

### Schema Overview

```sql
-- Core tables

users (
  id INTEGER PRIMARY KEY,
  name TEXT,                    -- stored encrypted
  email_hash TEXT,              -- SHA-256 hash for lookup, never plaintext
  created_at DATETIME,
  profile_type TEXT             -- 'primary' | 'family_member'
)

brokers (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  domain TEXT NOT NULL,
  removal_tier INTEGER,         -- 1=automated, 2=email, 3=manual
  opt_out_url TEXT,
  email_template TEXT,
  last_verified DATETIME,
  region TEXT,                  -- 'US' | 'EU' | 'UK' | 'global'
  active BOOLEAN DEFAULT TRUE
)

exposures (
  id INTEGER PRIMARY KEY,
  user_id INTEGER REFERENCES users(id),
  broker_id INTEGER REFERENCES brokers(id),
  first_seen DATETIME,
  last_seen DATETIME,
  data_fields TEXT,             -- JSON: which fields were found (name, address, phone, etc.)
  status TEXT                   -- 'found' | 'removal_pending' | 'removed' | 'reappeared'
)

removal_requests (
  id INTEGER PRIMARY KEY,
  exposure_id INTEGER REFERENCES exposures(id),
  tier INTEGER,
  submitted_at DATETIME,
  method TEXT,                  -- 'playwright' | 'email' | 'manual'
  status TEXT,                  -- 'pending' | 'submitted' | 'confirmed' | 'failed'
  next_check_at DATETIME
)

proof_records (
  id INTEGER PRIMARY KEY,
  removal_request_id INTEGER REFERENCES removal_requests(id),
  timestamp DATETIME NOT NULL,
  screenshot_path TEXT,         -- path to encrypted screenshot file
  http_response_code INTEGER,
  confirmation_text TEXT,       -- extracted confirmation message from page
  checksum TEXT                 -- SHA-256 of screenshot for tamper-evidence
)

scan_history (
  id INTEGER PRIMARY KEY,
  user_id INTEGER REFERENCES users(id),
  started_at DATETIME,
  completed_at DATETIME,
  brokers_checked INTEGER,
  exposures_found INTEGER,
  removals_initiated INTEGER,
  scan_type TEXT                -- 'full' | 'spot_check' | 'reverification'
)
```

### Backup Strategy

- `traceburn backup --output /path/to/backup.ctbak` — exports encrypted archive
- Archive format: AES-256-GCM encrypted tar.gz containing the SQLite file and screenshot proofs
- User controls backup location entirely — cloud, USB drive, NAS, their choice
- Backup does not require internet access

### What is NEVER Stored

| Prohibited | Why |
|-----------|-----|
| Plaintext PII in log files | Logs are often readable by other processes and retained indefinitely |
| Plaintext PII in the database | All sensitive fields encrypted at application layer before insert |
| Cloud sync of any kind | No opt-out dark patterns; user data never leaves device without explicit export |
| API keys or credentials in code | All secrets via environment variables or encrypted config file |
| Browser session cookies after removal | Playwright context cleared after each broker interaction |

## 4. Removal Workflow Decision

### Recommendation: Implement All 3 Tiers, Auto-Select by Broker Capability Map

### Tier Breakdown

| Tier | Method | Coverage | Implementation |
|------|--------|----------|----------------|
| **Tier 1** | Fully automated via Playwright | ~40% of brokers | Playwright fills and submits opt-out forms headlessly; screenshot captured as proof; confirmation text extracted |
| **Tier 2** | Email opt-out | ~30% of brokers | Jinja2-rendered opt-out email sent via user's SMTP (or local `sendmail`); email ID logged; delivery status tracked |
| **Tier 3** | Manual-assist wizard | ~30% of brokers | TraceBurn opens the opt-out URL in the user's default browser, displays step-by-step instructions in the terminal, and waits for user confirmation before marking complete |

### Decision Rationale

- Not all brokers expose automatable opt-out flows. Some require CAPTCHA, phone verification, or ID upload — these must remain Tier 3.
- Email opt-out (Tier 2) is fully automatable and covers a substantial fraction of brokers (especially older directory sites).
- Tier 1 automation covers the highest-volume, highest-risk brokers (Spokeo, Whitepages, BeenVerified, etc.).
- The broker capability map (`brokers.yaml`) determines which tier applies. This file is version-controlled and community-updatable.

### Broker Capability Map Entry Example

```yaml
- name: Spokeo
  domain: spokeo.com
  removal_tier: 1
  opt_out_url: https://www.spokeo.com/optout
  region: US
  requires_email_verification: true
  playwright_script: spokeo_optout.py
```

### Post-Removal Re-Verification

All removal requests are re-checked at:
- **7 days** — confirm removal processed
- **30 days** — verify data has not reappeared
- **90 days** — long-term persistence check

If data reappears, TraceBurn automatically re-initiates removal and alerts the user.
