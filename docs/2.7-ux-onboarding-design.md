## Overview

TraceBurn's second major differentiator is time-to-value. Aura users report 2-day setup processes, broken onboarding flows, and hours on support calls. TraceBurn's target is: **full onboarding completed in under 5 minutes, with zero support interaction, zero account creation, and zero prior knowledge required.**

This document specifies the `traceburn init` wizard interaction script, multi-profile design, international mode, plain-language status display, and error message standards.

### Wizard Implementation Notes

- **Input validation is inline.** Email format validated using a regex before accepting; phone validated against E.164 format (`^\+[1-9]\d{1,14}$`). If validation fails, the prompt repeats with a plain-language error (see Section 5).
- **Middle name and suffix are optional** with clear Enter-to-skip instructions. The name is stored as structured fields (first, middle, last, suffix) and also as a set of display-name variants for broker matching.
- **Password strength** is evaluated using `zxcvbn` (offline, no network call). Score of 3 or 4 required. If weak, the user is told why ("Too short", "Too common a word") and the prompt repeats.
- **Passphrase confirmation** uses `getpass` — characters are not echoed. The vault is not created until both prompts match.
- **Telegram step** can be skipped at `[y/N]` — notifications are optional. The connection step is skipped entirely; no error if the user says N.
- **Playwright install** uses `playwright install chromium` subprocess. Progress is piped to a Rich progress bar.
- **Target time: under 5 minutes.** The setup timer starts on first keypress and is displayed at completion. If Playwright download is slow, it is the only step likely to exceed 2 minutes; the progress bar communicates this explicitly.

## 2. Multi-Profile Support Design

### Profile Types

| Type | Description | Scan Access | Proof Store | Removal Queue |
|------|-------------|-------------|-------------|---------------|
| `primary` | The account owner | Full | Independent | Independent |
| `family_member` | Person explicitly added by primary user | Full | Independent | Independent |
| `monitored_only` | Person being monitored but not actively managed | Scan only | Shared with primary | None — alerts only |

Each profile has its own isolated PII vault entry, scan history, removal queue, and proof store directory. Profiles share the same encrypted SQLite database but no data crosses profile boundaries in queries or exports.

### Adding a Family Member

```
$ traceburn profiles add

explicit permission to scan and remove their data.

  Profile type:
    [1] Family member (full management — scan, remove, proof)
    [2] Monitored only (scan and alerts, no removals)

  Choose [1/2]: [1]

  ─────────────────────────────────────────────────
  Step 1 — Name
  ─────────────────────────────────────────────────
  First name: [Tom]
  Last name: [Smith]
  ... (same flow as traceburn init, Steps 1–7)

  ─────────────────────────────────────────────────
  Profile Summary
  ─────────────────────────────────────────────────
  Name:     Tom Smith
  Type:     Family member
  DOB:      March 3, 2006 (age 18)
  Vault:    Encrypted alongside primary profile

  ✓ Profile created. Run `traceburn scan --profile tom-smith` to scan.
```

### `traceburn profiles list`

```
$ traceburn profiles list

  Profiles
  ────────────────────────────────────────────────────────────────────────────
  Name            Type            Last Scan        Exposures    Active Removals
  ────────────────────────────────────────────────────────────────────────────
  Jane Smith      primary         2024-03-15        2 found      1 in progress
  Tom Smith       family member   2024-03-10        0 found      0

  ────────────────────────────────────────────────────────────────────────────

  Run `traceburn profiles add` to add a new profile.
```

### Lifecycle Event Alerts

TraceBurn tracks `date_of_birth` for all profiles with a DOB set and surfaces lifecycle alerts when relevant. These are not silent — they appear in `traceburn status` and as Telegram notifications if configured.

**Age-18 transition:**
```
[ALERT] Profile 'Tom Smith' is turning 18 on March 3, 2024.
        Family member profiles continue unchanged after age 18.
        No action required unless you want to remove their profile.
        Run `traceburn profiles review --profile tom-smith` to review.
```

**No profile is ever silently modified or removed.** Unlike Aura, which silently drops family members who turn 18, TraceBurn surfaces the event and requires explicit user action for any change.

### Profile Targeting in All Commands

Every TraceBurn command that operates on a profile accepts `--profile <name>`:

```
traceburn scan --profile tom-smith
traceburn remove --profile tom-smith
traceburn status --profile tom-smith
traceburn proof --export --profile tom-smith
traceburn history --profile tom-smith
```

If `--profile` is omitted, the primary profile is used.

## 3. International Mode Design

### Region-Scoped Scanning

```
traceburn scan                     # US brokers only (default)
traceburn scan --region EU         # EU-tagged brokers from brokers.yaml
traceburn scan --region UK         # UK brokers + GDPR template flag
traceburn scan --region global     # All regions
traceburn scan --region EU,US      # Multiple regions
```

Region tags are defined in `brokers.yaml` per broker entry:
```yaml
- name: Spokeo
  domain: spokeo.com
  region: US
  ...

- name: Pipl
  domain: pipl.com
  region: global
  ...

- name: 192.com
  domain: 192.com
  region: UK
  ...
```

### GDPR Article 17 Email Template

When `--region EU` or `--region UK` is used, Tier 2 removals for EU/UK-region brokers automatically use the GDPR Article 17 erasure request template instead of the standard CCPA template.

The template is rendered via Jinja2 from the letter specified in 1.4-legal-compliance.md:

```jinja2

limited to: {{ user.full_name }}, {{ user.email }},
{% if user.address %}{{ user.address }},{% endif %}
{% if user.phone %}{{ user.phone }}{% endif %}.

I request erasure on the following grounds:
[x] The data is no longer necessary for the purpose for which it was collected.
[ ] I withdraw the consent upon which processing was based.
[ ] I object to the processing under Article 21 GDPR.

```

### UK ICO Complaint Template

For UK-region brokers that fail to respond within 30 days of a GDPR erasure request, TraceBurn generates an ICO complaint template:

```
$ traceburn remove --broker 192.com --escalate

192.com has not responded to your erasure request submitted 2024-02-14
(31 days ago). UK GDPR requires a response within 30 days.

TraceBurn has prepared an ICO complaint for you.

  Generated: ~/Downloads/ico_complaint_192com_20240315.txt

  To file:
    1. Go to ico.org.uk/make-a-complaint
    2. Select "Data protection complaint"
    3. Attach the generated file or copy its contents

  Do you want to open the ICO complaints page now? [y/N]:
```

### Address and Phone Format Handling

All address and phone fields accept international formats:

- **Phone:** E.164 format required (`+[country code][number]`). The prompt shows examples for the user's detected locale if available.
- **Address:** Free-form street + city + postal code + country. Country defaults to US but can be any ISO 3166-1 alpha-2 code. No US-specific field labels ("State" becomes "State / Province / Region").
- **Date of birth:** ISO 8601 (`YYYY-MM-DD`) always. No MM/DD/YYYY ambiguity.

TraceBurn never assumes US-only. The word "State" never appears alone in a prompt without "/ Province" alongside it.

### Status Display Rules

1. **No score.** No "Privacy Score: 72/100." Scores obscure specifics and invite gaming.
2. **No fear language.** No "CRITICAL EXPOSURE DETECTED." No "YOUR IDENTITY IS AT RISK."
3. **No vague reassurance.** No "we're on it" without concrete details of what is happening and when.
4. **Every status item is specific.** Which broker, what data, what action was taken, when the next event is.
5. **One CTA at the bottom.** Not a menu of options. A single, unambiguous next step.
6. **Reappearance is contextualized, not alarming.** "This is common — data brokers re-collect information from public records automatically." The user learns something; they do not panic.
7. **Dates are human-readable.** "March 20, 2024" not "2024-03-20T09:44:01Z." ISO 8601 is for machine interfaces; `--format json` provides it.

## 5. Error Messages & Edge Cases

### Design Principles

Every error message must answer three questions:
1. **What happened?** (Plain English, no stack trace)
2. **Why did it happen?** (Cause the user can understand)
3. **What do they do next?** (Specific, actionable)

Stack traces are written to the audit log at `~/.traceburn/logs/traceburn.log` and never shown to the user. If the user wants the trace, they can pass `--debug`.

### Error: Wrong Vault Passphrase

```
  ✗ Incorrect passphrase

    The passphrase you entered does not match the one used to
    create your vault.

    If you have forgotten your passphrase, your vault cannot be
    recovered — this is by design. Your data is encrypted in a
    way that makes recovery without the passphrase impossible.

    If you have a backup of your vault from before you changed
    your passphrase, you can restore it:
      traceburn restore --from /path/to/backup.ctbak

    If you believe you know the correct passphrase, try again:
      (3 attempts remaining before a 60-second cooldown)
```

*Note: After 5 failed attempts, TraceBurn introduces a 60-second delay between attempts. This is implemented in software — it is not a lockout. The delay discourages brute-force guessing but does not lock the user out of their own data.*

### Audit Log Standard

All errors write structured log entries to `~/.traceburn/logs/traceburn.log`:

```
2024-03-15T09:42:17Z ERROR broker=spokeo event=CAPTCHA_DETECTED url=https://www.spokeo.com/optout tier=1
2024-03-15T09:44:00Z ERROR broker=beenverified event=FORM_CHANGED selector=".optout-submit-btn" http_code=200
2024-03-15T09:45:33Z ERROR event=TELEGRAM_UNREACHABLE http_code=None retry_at=2024-03-15T09:55:33Z
```

Log entries contain:
- UTC ISO 8601 timestamp
- Log level (ERROR, WARN, INFO)
- Structured key=value fields (no PII in logs)
- Exception type and message (no stack trace in standard log; full stack trace if `--debug`)

PII (names, addresses, emails, phone numbers) is **never written to log files**, even in debug mode.

## Design Rationale Summary

| Decision | Choice | Why |
|----------|--------|-----|
| 5-minute target for init | Hard requirement | Directly addresses Aura Gap 5; measured and displayed at completion |
| Single CTA at status bottom | One action, not a menu | Menus introduce choice paralysis; one clear next step is always better than four vague options |
| No score or rating | Omitted by design | Scores obscure specifics, invite gaming, and generate anxiety without information |
| Reappearance contextualized | Explain, don't alarm | Users who understand why data reappears are more patient and less likely to abandon the tool |
| Error messages include "what to do next" | Required field | Errors without next steps generate support requests (Aura Gap 5) — TraceBurn has no support |
| Stack traces to log, not terminal | Log only | Non-technical users cannot act on a stack trace; it degrades trust; developers use `--debug` |
| International format acceptance | Required | Directly addresses Aura Gap 6; US-only input formats exclude EU and UK users |
| Lifecycle alerts always explicit | No silent changes | Directly addresses Aura Gap 8 — family plan silent drops |
