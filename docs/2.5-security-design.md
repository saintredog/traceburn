## 1. PII Vault Encryption

### 1.1 Design Overview

The PII vault protects all sensitive user data at rest using application-layer encryption. Every sensitive field is encrypted individually before being written to the SQLite database. The encryption key is derived at runtime from the user's master passphrase and is **never written to disk**.

**Encryption scheme:**
- Key derivation: PBKDF2HMAC (SHA-256, 600,000 iterations, 32-byte random salt per field)
- Cipher: AES-256-GCM (authenticated encryption — provides both confidentiality and integrity)
- Salt storage: 32-byte salt stored as a prefix to the ciphertext blob in the database column
- Nonce storage: 12-byte GCM nonce stored as a prefix after the salt
- Tag: 16-byte GCM authentication tag appended by the `cryptography` library automatically

**On-disk blob layout per encrypted field:**

```
[32-byte salt][12-byte nonce][16-byte tag + N-byte ciphertext]
```

The key is derived fresh for every encrypt/decrypt operation from `salt + passphrase`. This means each field has its own independent salt and therefore its own unique derived key, even when encrypted with the same passphrase.

### 1.2 Python Implementation

```python
# src/vault.py

import os
import secrets
import ctypes
from typing import Union

from cryptography.hazmat.primitives.kdf.pbkdf2 import PBKDF2HMAC
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.ciphers.aead import AESGCM


PBKDF2_ITERATIONS = 600_000
SALT_LENGTH = 32    # bytes
NONCE_LENGTH = 12   # bytes (96-bit, GCM standard)
KEY_LENGTH = 32     # bytes (256-bit AES key)


def _derive_key(passphrase: Union[str, bytes], salt: bytes) -> bytes:
    """
    Derive a 256-bit AES key from passphrase + salt using PBKDF2HMAC-SHA256.
    Key is returned as bytes and NEVER stored to disk.
    """
    if isinstance(passphrase, str):
        passphrase = passphrase.encode("utf-8")

    kdf = PBKDF2HMAC(
        algorithm=hashes.SHA256(),
        length=KEY_LENGTH,
        salt=salt,
        iterations=PBKDF2_ITERATIONS,
    )
    return kdf.derive(passphrase)


def encrypt(plaintext: Union[str, bytes], passphrase: str) -> bytes:
    """
    Encrypt a plaintext value with AES-256-GCM.

    Returns a single blob: [salt (32)] + [nonce (12)] + [ciphertext+tag]
    The passphrase is NOT stored. The returned blob contains everything
    needed to decrypt with the correct passphrase.

    Memory safety: plaintext bytes are zeroed after encryption.
    """
    if isinstance(plaintext, str):
        plaintext_bytes = plaintext.encode("utf-8")
    else:
        plaintext_bytes = bytes(plaintext)

    salt = secrets.token_bytes(SALT_LENGTH)
    nonce = secrets.token_bytes(NONCE_LENGTH)

    key = _derive_key(passphrase, salt)
    aesgcm = AESGCM(key)

    try:
        ciphertext = aesgcm.encrypt(nonce, plaintext_bytes, associated_data=None)
    finally:
        # Zero out sensitive material from memory
        _zero_bytes(key)
        _zero_bytes(plaintext_bytes)

    return salt + nonce + ciphertext


def decrypt(blob: bytes, passphrase: str) -> str:
    """
    Decrypt a blob produced by encrypt().

    Raises ValueError if the passphrase is wrong or the ciphertext is tampered
    (GCM authentication tag mismatch raises cryptography.exceptions.InvalidTag,
    which is caught and re-raised as ValueError to avoid leaking implementation details).
    """
    if len(blob) < SALT_LENGTH + NONCE_LENGTH + 16:
        raise ValueError("Ciphertext blob is too short to be valid.")

    salt = blob[:SALT_LENGTH]
    nonce = blob[SALT_LENGTH:SALT_LENGTH + NONCE_LENGTH]
    ciphertext = blob[SALT_LENGTH + NONCE_LENGTH:]

    key = _derive_key(passphrase, salt)
    aesgcm = AESGCM(key)

    try:
        plaintext_bytes = aesgcm.decrypt(nonce, ciphertext, associated_data=None)
    except Exception:
        raise ValueError("Decryption failed: wrong passphrase or data corrupted.")
    finally:
        _zero_bytes(key)

    plaintext = plaintext_bytes.decode("utf-8")
    _zero_bytes(plaintext_bytes)
    return plaintext


def _zero_bytes(data: Union[bytes, bytearray]) -> None:
    """
    Overwrite a bytes or bytearray object with zeros to reduce time-in-memory
    of sensitive values.

    Note: CPython does not guarantee this defeats all in-memory forensics
    (e.g., the GC may have copied the object). This is best-effort.
    """
    if isinstance(data, bytearray):
        for i in range(len(data)):
            data[i] = 0
    else:
        # bytes is immutable; use ctypes to overwrite the backing buffer
        try:
            buf = (ctypes.c_char * len(data)).from_buffer_copy(data)
            ctypes.memset(buf, 0, len(data))
        except TypeError:
            pass  # Cannot zero — log at debug level, continue
```

### 1.3 Passphrase Change (Re-key Procedure)

When the user runs `traceburn vault --rekey`:

```python
def rekey_vault(db, old_passphrase: str, new_passphrase: str) -> None:
    """
    Re-encrypt all encrypted fields in the database with a new passphrase.

    Steps:
    1. Decrypt every encrypted field using old_passphrase
    2. Re-encrypt each field using new_passphrase (new salt + nonce generated per field)
    3. Write all new blobs to the database in a single transaction
    4. Zero plaintext buffers after each field

    If any step fails, the transaction is rolled back. The database is never
    left in a partially re-keyed state.
    """
    encrypted_fields = db.get_all_encrypted_fields()  # returns list of (table, row_id, column, blob)
    updates = []

    for table, row_id, column, old_blob in encrypted_fields:
        plaintext = decrypt(old_blob, old_passphrase)
        new_blob = encrypt(plaintext, new_passphrase)
        updates.append((table, row_id, column, new_blob))
        # plaintext is zeroed inside encrypt() above

    db.apply_rekey_updates(updates)  # single atomic transaction
```

## 2. Config File Access Controls

### 2.1 Permission Matrix

| Path | Permissions | Octal | Rationale |
|------|-------------|-------|-----------|
| `~/.traceburn/user.yaml` | owner r/w only | `600` | Contains PII profile and notification config |
| `~/.traceburn/traceburn.db` | owner r/w only | `600` | Encrypted SQLite; no other user should open it |
| `~/.traceburn/proofs/` | owner r/w/x only | `700` | Screenshot proof directory; no read by others |
| `~/.traceburn/audit.log` | see section 3 | `444` (between writes) | Append-only audit log |
| `~/.traceburn/` (root dir) | owner r/w/x only | `700` | Entire data directory private |

### 2.2 install.sh Enforcement

```bash
#!/usr/bin/env bash
# install.sh — TraceBurn installer (relevant security section)

set -euo pipefail

CLEARTRACE_DIR="$HOME/.traceburn"
PROOFS_DIR="$CLEARTRACE_DIR/proofs"
CONFIG_FILE="$CLEARTRACE_DIR/user.yaml"
DB_FILE="$CLEARTRACE_DIR/traceburn.db"

# Create directory structure with correct permissions
mkdir -p "$CLEARTRACE_DIR"
chmod 700 "$CLEARTRACE_DIR"

mkdir -p "$PROOFS_DIR"
chmod 700 "$PROOFS_DIR"

# If user.yaml does not exist, create from example
if [ ! -f "$CONFIG_FILE" ]; then
    cp config/user.yaml.example "$CONFIG_FILE"
fi
chmod 600 "$CONFIG_FILE"

# If DB does not exist, it will be created on first run by db.py
# Pre-enforce permissions if the file exists
if [ -f "$DB_FILE" ]; then
    chmod 600 "$DB_FILE"
fi

echo "Permissions enforced:"
ls -la "$CLEARTRACE_DIR"
```

TraceBurn also enforces permissions at runtime on every startup:

```python
# src/config.py

import os
import stat
import logging

logger = logging.getLogger(__name__)

REQUIRED_FILE_MODE = 0o600
REQUIRED_DIR_MODE = 0o700

def check_file_permissions(path: str) -> None:
    """Warn if a sensitive file has permissions more open than required."""
    mode = stat.S_IMODE(os.stat(path).st_mode)
    if mode & 0o077:  # any group or other bits set
        logger.warning(
            "SECURITY: %s has permissions %o — expected %o. "
            "Run: chmod 600 %s",
            path, mode, REQUIRED_FILE_MODE, path
        )
```

## 3. Audit Log Design

### 3.1 Log Format

Every action TraceBurn takes is appended to `~/.traceburn/audit.log`.

**Format:** `ISO_TIMESTAMP | ACTION_TYPE | BROKER | STATUS | RECORD_ID`

```
2026-04-01T09:02:11Z | SCAN_STARTED       | ALL_BROKERS        | started   | scan:a3f8c1d2
2026-04-01T09:02:18Z | BROKER_SCANNED     | spokeo.com         | found     | exposure:441
2026-04-01T09:02:19Z | BROKER_SCANNED     | beenverified.com   | found     | exposure:442
2026-04-01T09:02:45Z | BROKER_SCANNED     | whitepages.com     | not_found | -
2026-04-01T09:14:32Z | SCAN_COMPLETE      | ALL_BROKERS        | success   | scan:a3f8c1d2
2026-04-01T09:14:33Z | REMOVAL_SUBMITTED  | spokeo.com         | submitted | removal:881
2026-04-01T09:14:34Z | REMOVAL_SUBMITTED  | beenverified.com   | submitted | removal:882
2026-04-01T09:21:00Z | PROOF_CAPTURED     | spokeo.com         | valid     | proof:1101
2026-04-01T09:21:01Z | PROOF_CAPTURED     | beenverified.com   | valid     | proof:1102
2026-03-17T09:02:11Z | REMOVAL_CONFIRMED  | spokeo.com         | confirmed | removal:881
2026-04-01T09:05:44Z | REAPPEARANCE       | beenverified.com   | reappeared| exposure:442
2026-04-01T09:05:45Z | REMOVAL_SUBMITTED  | beenverified.com   | submitted | removal:903
2026-04-01T09:06:00Z | NOTIFICATION_SENT  | TELEGRAM           | success   | -
```

**PII policy:** Log entries contain only:
- Broker domain names (not user data)
- Internal record IDs (opaque integers or UUIDs)
- Action types and status values (enumerated)
- Timestamps

Log entries **never** contain: name, address, phone, email, DOB, SSN, or any other user PII.

### 3.2 Append-Only Enforcement

The audit log uses a write pattern that minimizes the window during which the file is writable:

```python
# src/audit.py

import os
import stat
import logging
from datetime import datetime, timezone
from pathlib import Path

AUDIT_LOG_PATH = Path.home() / ".traceburn" / "audit.log"


def log_action(
    action_type: str,
    broker: str,
    status: str,
    record_id: str = "-",
) -> None:
    """
    Append a single audit log entry. No PII may appear in any parameter.

    Thread-safe via file append mode (atomic on POSIX for small writes).
    """
    timestamp = datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%SZ")
    entry = f"{timestamp} | {action_type:<20} | {broker:<20} | {status:<12} | {record_id}\n"

    path = str(AUDIT_LOG_PATH)

    # Temporarily make writable (644), write, restore append-only (444)
    try:
        if AUDIT_LOG_PATH.exists():
            os.chmod(path, 0o644)

        with open(path, "a", encoding="utf-8") as f:
            f.write(entry)

    finally:
        os.chmod(path, 0o444)  # restore read-only even if write failed
```

### 3.3 Log Rotation

Log rotation is configured via Python's `logging.handlers.RotatingFileHandler`:

```python
import logging
from logging.handlers import RotatingFileHandler

def configure_audit_logging() -> None:
    handler = RotatingFileHandler(
        filename=str(AUDIT_LOG_PATH),
        maxBytes=10 * 1024 * 1024,  # 10 MB max per file
        backupCount=5,               # keep 5 rotated files: audit.log.1 through .5
        encoding="utf-8",
    )
    handler.setFormatter(logging.Formatter("%(message)s"))  # pre-formatted entries

    audit_logger = logging.getLogger("traceburn.audit")
    audit_logger.addHandler(handler)
    audit_logger.setLevel(logging.INFO)
    audit_logger.propagate = False  # don't send audit entries to root logger
```

Rotated files: `audit.log`, `audit.log.1`, `audit.log.2`, ... `audit.log.5`
All rotated files receive `chmod 444` after rotation.

## 4. Credential Management

### 4.1 Environment Variable Model

All secrets are provided via environment variables. TraceBurn never stores secrets in plaintext files, code, or the database.

| Variable | Required | Purpose |
|----------|----------|---------|
| `CLEARTRACE_VAULT_PASS` | **Required** | Master passphrase for AES-256-GCM vault key derivation. Never stored. |
| `TELEGRAM_BOT_TOKEN` | Optional | Telegram Bot API token for notifications |
| `TELEGRAM_CHAT_ID` | Optional | Destination Telegram chat ID |
| `HIBP_API_KEY` | Optional | Have I Been Pwned API key for breach lookup |
| `DEHASHED_API_KEY` | Optional | DeHashed API key for breach database queries |

### 4.2 Startup Validation

```python
# src/config.py

import os
import sys


REQUIRED_ENV_VARS = ["CLEARTRACE_VAULT_PASS"]
OPTIONAL_ENV_VARS = ["TELEGRAM_BOT_TOKEN", "TELEGRAM_CHAT_ID", "HIBP_API_KEY", "DEHASHED_API_KEY"]


def validate_environment() -> None:
    """
    Validate that all required environment variables are present.
    Called at CLI startup before any other operation.

    Exits with a clear error message if any required var is missing.
    Does NOT print the values of env vars.
    """
    missing = [var for var in REQUIRED_ENV_VARS if not os.environ.get(var)]

    if missing:
        print("ERROR: The following required environment variables are not set:")
        for var in missing:
            print(f"  - {var}")
        print()
        print("Set them before running TraceBurn:")
        for var in missing:
            print(f"  export {var}='your_value_here'")
        print()
        print("For CLEARTRACE_VAULT_PASS, choose a strong, unique passphrase.")
        print("TraceBurn will use it to encrypt and decrypt your PII vault.")
        sys.exit(1)

    optional_present = [var for var in OPTIONAL_ENV_VARS if os.environ.get(var)]
    if optional_present:
        # Log which optional features are enabled — not the values
        import logging
        logger = logging.getLogger(__name__)
        logger.debug("Optional features enabled: %s", ", ".join(optional_present))
```

### 4.4 Secret Rotation Procedure

**Changing the vault passphrase:**

```bash
# Step 1: set both old and new passphrases in environment
export CLEARTRACE_VAULT_PASS="old-passphrase"
export CLEARTRACE_VAULT_PASS_NEW="new-passphrase"

# Step 2: run the re-key command
traceburn vault --rekey

# Step 3: update your shell environment with the new passphrase
export CLEARTRACE_VAULT_PASS="new-passphrase"
unset CLEARTRACE_VAULT_PASS_NEW
```

The `--rekey` command:
1. Decrypts every encrypted field with the old passphrase
2. Re-encrypts each field with a fresh salt + nonce using the new passphrase
3. Writes all updates in a single atomic database transaction
4. Zeroes all plaintext buffers after use
5. On success, logs `VAULT_REKEYED | - | success | -` to audit log (no passphrases in log)

**Rotating Telegram bot token:**
```bash
# Revoke old token via BotFather (/revoke), get new token, update env var
export TELEGRAM_BOT_TOKEN="new_token"
traceburn notify --test
```

## 5. Threat Model

### 5.1 Threat: Local File Access by Another System User

**Scenario:** Another user on the same machine (or a compromised process running as a different user) attempts to read TraceBurn data files.

**Mitigations:**
1. `~/.traceburn/` has permissions `700` — no group or world read/execute
2. `user.yaml` and `traceburn.db` have permissions `600` — no group or world access
3. Even if file permissions were bypassed (e.g., root access), the SQLite database fields are AES-256-GCM encrypted — raw file contents are ciphertext only
4. The vault key is derived at runtime from the passphrase, which is never stored on disk

**Residual risk:** A root-level attacker can bypass POSIX permissions. Encrypt the home directory (FileVault on macOS, LUKS on Linux) for full-disk protection.

---

### 5.2 Threat: Network Interception of Scan Traffic

**Scenario:** An attacker on the local network (or a compromised router) intercepts TraceBurn's HTTP requests to data broker sites, potentially learning which brokers are being queried and what search terms are sent.

**Mitigations:**
1. All broker requests use HTTPS exclusively — enforced in `broker_client.py` by rejecting any opt-out URL not prefixed with `https://`
2. Playwright uses the system's trusted CA store for certificate validation; invalid certificates cause an immediate abort, not a warning
3. Certificate pinning is considered but not implemented in v1 (broker certificates rotate frequently; pinning would cause false failures). Revisit in v2 if a broker is deemed high-risk enough to warrant it.
4. Users on untrusted networks should run TraceBurn over a trusted VPN

**Residual risk:** The fact that TraceBurn is querying a specific broker (e.g., Spokeo) leaks metadata even over HTTPS. DNS-over-HTTPS or Tor would mitigate this but are out of scope for v1.

---

### 5.3 Threat: Malicious Broker Opt-Out Page (XSS in Playwright Context)

**Scenario:** A malicious or compromised broker opt-out page attempts to execute JavaScript that exfiltrates data from the browser context, reads clipboard content, or escapes the browser sandbox.

**Mitigations:**
1. **Isolated browser context per broker:** A new Playwright `BrowserContext` is created for each broker interaction and destroyed after it completes. No cookies, local storage, or session data persists between brokers.
2. **No shared credentials in browser context:** TraceBurn never loads passwords or API keys into the browser environment. The browser context contains only what is needed for that specific opt-out form.
3. **No persistent browser profile:** Playwright uses `browser.new_context()` (ephemeral), not a persistent user data directory.
4. **Strict navigation:** Playwright navigation is limited to the expected opt-out URL domain. Redirects to unexpected domains are blocked via `page.route()` interceptors.
5. **No JavaScript evaluation of untrusted content:** TraceBurn does not call `page.evaluate()` with user-controlled strings.

```python
# Per-broker isolation pattern (tier1_playwright.py)
async with browser.new_context(
    java_script_enabled=True,  # required for form submission
    accept_downloads=False,     # block file downloads
    bypass_csp=False,           # respect Content-Security-Policy
) as context:
    page = await context.new_page()
    await page.goto(broker.opt_out_url)
    # ... form submission logic ...
# Context and all its state are destroyed when the `async with` block exits
```

**Residual risk:** A browser 0-day exploit could escape the sandbox. Chromium's sandbox provides process isolation at the OS level; this is the last line of defence and is outside TraceBurn's control.

---

### 5.4 Threat: Audit Log Leaking PII

**Scenario:** A developer adds a log statement that includes a user's name, address, or phone number, and this appears in `audit.log`.

**Mitigations:**
1. **PII-free logging policy** enforced in code review: audit log entries may contain only broker domain names, internal record IDs, action types, and status values — never user-supplied data
2. The `log_action()` function accepts only enumerated parameters — it cannot accept free-form strings that might contain PII
3. Regular log audits: `grep -E '[0-9]{3}-[0-9]{4}|@[a-z]+\.[a-z]' ~/.traceburn/audit.log` in CI to catch accidental PII leakage
4. The `audit.log` file itself has permissions `444` (read-only between writes), limiting which processes can append to it

**Residual risk:** Python's `logging` module can be accidentally misconfigured to capture stack traces containing variable values. Suppress stack traces in the audit logger specifically.

---

### 5.5 Threat: Backup File Compromise

**Scenario:** A `.ctbak` backup file is stolen (e.g., from cloud storage, email, or a shared drive) and an attacker attempts to extract PII from it.

**Mitigations:**
1. Backup files are encrypted with the same AES-256-GCM vault key derived from the user's passphrase — an attacker without the passphrase sees only ciphertext
2. Backup format: encrypted tar.gz wrapping the SQLite database and proof screenshots — no plaintext metadata in the archive
3. `.ctbak` files are gitignored and excluded from all sync paths by default
4. Backup passphrase is the same as the vault passphrase — no separate credential to manage (single secret)

**Residual risk:** A weak or reused passphrase makes the backup vulnerable to brute-force. Users are instructed to use a long, random passphrase (minimum 16 characters, ideally a random diceware phrase).

---

### 5.6 Threat: Supply Chain Attack on Dependencies

**Scenario:** A malicious package update to one of TraceBurn's PyPI dependencies (e.g., `requests`, `playwright`, `cryptography`) introduces backdoor code that exfiltrates vault contents or passphrase.

**Mitigations:**
1. **Pinned requirements with hashes** — `requirements.txt` uses `pip install --require-hashes`:
   ```
   cryptography==42.0.8 \
       --hash=sha256:abcdef1234567890... \
       --hash=sha256:fedcba0987654321...
   requests==2.32.3 \
       --hash=sha256:...
   ```
2. **Dependency pinning at exact versions** — no floating `>=` in production requirements; `requirements-dev.txt` may use ranges for development only
3. **Minimal dependency surface** — TraceBurn avoids unnecessary dependencies; each package must justify its inclusion
4. **Dependabot / pip-audit in CI** — automated vulnerability scanning on every PR; CVE alerts trigger immediate patch
5. **Reproducible installs** — `install.sh` uses `pip install --require-hashes -r requirements.txt` to ensure hash verification at install time

```bash
# Generate pinned requirements with hashes
pip-compile --generate-hashes requirements.in -o requirements.txt

# Install with hash verification
pip install --require-hashes -r requirements.txt
```

**Residual risk:** A zero-day in a dependency could be exploited before hash pinning is updated. The attack window is limited to the time between exploit publication and the next `pip-audit` run.
