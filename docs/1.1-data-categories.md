# 1.1 — Data Categories to Scan

> **TraceBurn Phase 1 Research**
> Created: 2026-04-01
> WBS Tasks: 1.1.1, 1.1.2, 1.1.3

---

## Summary (Key Findings)

- **10 personal identifier categories** are defined for scanning, ranging from SSNs (highest sensitivity) to IP addresses (lowest).
- **Three match tiers** cover exact, partial, and fuzzy detection — with confidence scores for scoring/prioritization in the removal engine.
- Most broker sites present phone numbers in multiple formats; the scanner must normalize all variants before comparison.
- Fuzzy matching is critical for names and addresses since brokers frequently use maiden names, middle initials, abbreviations, and common misspellings.
- SSN partials require special handling — only the last 4 digits typically appear publicly, but even those are high-sensitivity.
- Re-scanning for all data types should be scheduled (see WBS 3.5) since brokers re-list after removal.

## 2. Sensitivity Ranking Table

| Identifier | Sensitivity | Rationale |
|---|---|---|
| SSN Partial | **HIGH** | Even last-4 combined with DOB + name enables identity theft / account takeover |
| Date of Birth | **HIGH** | Frequently used as a verification factor; rarely changes |
| Home Address | **HIGH** | Physical safety risk; enables stalking, doxxing, targeted physical crime |
| Full Name | **HIGH** | Primary linkage key that ties all other data together |
| Phone Numbers | **HIGH** | Enables SIM-swap attacks, phishing, and harassment |
| Email Addresses | **HIGH** | Enables phishing, credential stuffing, account recovery attacks |
| Relatives' Names | **MEDIUM** | Exposes family members to secondary targeting; used as broker profile linkage |
| Employer | **MEDIUM** | Enables spear-phishing and social engineering; can affect employment |
| Usernames / Handles | **MEDIUM** | Cross-platform correlation risk; can link anonymous activity to real identity |
| IP Addresses | **LOW** | Geolocation only (city-level); changes frequently; limited direct harm |

**Severity definitions:**
- **HIGH** — Direct path to identity theft, financial fraud, physical harm, or account compromise.
- **MEDIUM** — Enables secondary attacks or privacy violations; harmful but typically requires additional data.
- **LOW** — Useful for profiling but limited standalone harm; still worth removing.

## 3. Match Definition Specification

### 3.1 Match Tiers

| Match Type | Definition | Confidence Score |
|---|---|---|
| **Exact** | Byte-for-byte identical after normalization (lowercased, whitespace-collapsed) | **1.0** |
| **Partial** | The identifier appears as a substring within a larger field | **0.7** |
| **Fuzzy** | Levenshtein distance ≤ 2 between identifier and candidate string | **0.5** |

### 3.2 Normalization Rules (Applied Before Matching)

All strings are normalized before comparison:
- Lowercase entire string
- Collapse internal whitespace to single space
- Strip leading/trailing whitespace
- Remove punctuation for phone/SSN comparison (strip `()`, `-`, `.`, spaces)
- Expand common abbreviations: `St.` → `Street`, `Ave.` → `Avenue`, `Dr.` → `Drive`

### 3.3 Exact Match

- Definition: After normalization, `candidate == identifier`
- Used for: Email addresses, phone numbers (after stripping formatting), SSN partials, IPs
- Example: Target email `clifford@example.com` matches page text `clifford@example.com` exactly

### 3.4 Partial Match

- Definition: After normalization, `identifier in candidate` (substring check)
- Used for: Names appearing within full profile strings, addresses appearing within paragraphs
- Example: Target `"clifford roberts"` found within `"clifford roberts iii, age 34, san diego ca"`
- Minimum substring length: 4 characters (to avoid false positives on short strings)

### 3.5 Fuzzy Match

- Algorithm: Levenshtein distance (edit distance)
- Threshold: Distance ≤ 2 edits (insertions, deletions, substitutions)
- Used for: Full names, street addresses (typos, abbreviations, variant spellings)
- NOT used for: Email, phone, SSN, IP (these must be exact or partial only — fuzzy on these generates too many false positives)
- Example: `"cliford roberts"` (1 deletion) matches `"clifford roberts"` — distance = 1, fuzzy match

### 3.6 Confidence Scoring

Confidence scores are used to prioritize findings in the removal queue:

```
confidence = 1.0   → Exact match → immediate removal action
confidence = 0.7   → Partial match → high-priority review + removal
confidence = 0.5   → Fuzzy match → medium-priority → human confirmation before removal
```

When multiple match types apply to the same finding, take the **highest** confidence score.

Composite scoring (for records with multiple matching fields):
```
composite_score = max(individual_confidence_scores) + 0.1 * (num_fields_matched - 1)
composite_score = min(composite_score, 1.0)  # cap at 1.0
```

## 4. How Each Identifier Appears on Broker Sites

| Identifier | Common Formats on Broker Sites | Normalization Notes |
|---|---|---|
| **Full Name** | "Clifford Roberts", "Roberts, Clifford", "CLIFFORD A ROBERTS III", "Cliff Roberts" | Lowercase; handle suffixes (Jr, Sr, III, IV) separately; check maiden/alternate names |
| **Email** | `clifford@example.com`, sometimes obfuscated as `c****d@example.com` | Exact match on unobfuscated; flag obfuscated for partial match |
| **Phone Numbers** | `(619) 555-1234`, `619-555-1234`, `6195551234`, `619.555.1234`, `+16195551234` | Strip all non-digits before comparison; store E.164 form internally |
| **Home Address** | `123 Main St, San Diego, CA 92101`, `123 Main Street San Diego California`, `123 Main St Apt 4B San Diego 92101` | Normalize abbreviations; split into components (street, city, state, zip) for partial matching |
| **Date of Birth** | `04/15/1988`, `April 15, 1988`, `1988-04-15`, age shown as `"Age 37"` or `"Age: 37-40"` | Match on full DOB or year + month; flag age ranges that include target's age |
| **SSN Partial** | `***-**-4821`, `xxxx4821`, `Last 4: 4821` | Match only on the 4-digit suffix; never log or store full SSN |
| **Usernames** | `@clifftrace`, `clifftrace`, `clifford_r`, shown in "Associated Accounts" sections | Exact and partial; some brokers surface social handles aggregated from public profiles |
| **IP Addresses** | `192.168.1.1` (rare on consumer brokers), more common on paste sites or breach dumps | Exact match only; check against known home IP and any noted VPN exit IPs |
| **Employer** | `"Works at: Acme Corp"`, `"Employer: Acme Corporation"`, `"Acme Corp, San Diego CA"` | Normalize company name variants; partial match on core company name |
| **Relatives' Names** | `"Relatives: Jane Roberts, Robert Roberts"`, listed as "Associated People" or "Known Associates" | Treat as linked identifiers; finding relative names confirms profile match even if primary name varies |

## 5. Special Handling Notes

### 5.1 Name Variants
The scanner must check all known name variants:
- Full legal name
- Common nickname (Cliff for Clifford)
- Name with/without suffix (III, Jr, Sr)
- Maiden name (if applicable)
- Middle initial included vs. omitted

### 5.2 Historical Addresses
Brokers aggregate address history, not just current address. The PII vault must store all prior addresses to scan against, not just the current one.

### 5.3 Phone Number History
Ported/recycled phone numbers may appear on broker profiles for prior owners. Only flag if combined with name match.

### 5.4 Age Range vs. Exact DOB
Many brokers display an age range (e.g., "Age 35-40") rather than exact DOB. The scanner should flag records where the target's age falls within the displayed range AND name/address also matches.

### 5.5 Relatives as Linkage Data
Even if a broker profile uses a slightly wrong name or old address, the presence of correct relative names is a strong signal that the profile belongs to the target. Relatives' names should boost composite confidence score.
