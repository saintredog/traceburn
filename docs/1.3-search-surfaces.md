**TraceBurn Phase 1 Research**

This document catalogs each surface where personal information can be exposed, including severity, remediation effort, and TraceBurn automation potential.

## 1. Google / Bing Index Exposure

### Overview
Search engine indexes cache and surface personal information from data brokers, public records, leaked documents, and third-party sites. Even after a broker removes a page, the cached version can persist in search results for weeks.

### Common Dork Patterns

```
# Data broker profile pages
site:spokeo.com "Firstname Lastname"
site:whitepages.com "Firstname Lastname"
site:intelius.com "Firstname Lastname"
site:beenverified.com "Firstname Lastname"
site:peoplefinder.com "Firstname Lastname"
site:mylife.com "Firstname Lastname"
site:instantcheckmate.com "Firstname Lastname"
site:truthfinder.com "Firstname Lastname"
site:radaris.com "Firstname Lastname"

# Document / PDF exposure
"Firstname Lastname" filetype:pdf
"Firstname Lastname" filetype:xls OR filetype:xlsx
"Firstname Lastname" filetype:doc OR filetype:docx

# Email + name combos
"firstname.lastname@gmail.com"
"user@olddomain.com" -site:linkedin.com

# Address exposure
"123 Main St" "Firstname Lastname"
"Firstname Lastname" "SSN" OR "social security"

# Phone number exposure
"555-867-5309" "Lastname"
```

### How Cached Pages Work

Google and Bing maintain crawl caches independently of the source site. When a broker removes a profile:

1. The broker's live page returns 404 or redirects.
2. Google's cache still serves the old snapshot (typically 2–6 weeks lag).
3. SERP snippets (the text preview under a search result) can remain visible even longer, drawn from the cached copy.
4. Bing cache is independent — Google removal does NOT trigger Bing removal.

**Key insight:** Removal from the broker does not equal removal from search. Both must be pursued independently.

### Removal Request URLs

| Engine | Tool | URL |
|--------|------|-----|
| Google | Remove Outdated Content | https://search.google.com/search-console/remove-outdated-content |
| Google | Search Console URL Removal (for verified owners) | https://search.google.com/search-console/removals |
| Google | Personal Info Removal Request | https://support.google.com/websearch/troubleshooter/9685456 |
| Bing | Content Removal Request | https://www.bing.com/webmaster/tools/contentremoval |
| Bing | Bing Webmaster Tools (for site owners) | https://www.bing.com/webmasters/ |

### Google "Remove Outdated Content" Workflow

1. Navigate to the Remove Outdated Content tool.
2. Enter the full URL of the cached page (not the broker's homepage).
3. Google checks if the page is truly gone (404/410) or has been substantially changed.
4. If confirmed, the cache is cleared within 1–3 days.
5. SERP snippet removal is separate — use the Personal Info Removal form for that.

**Important:** Google's Personal Info Removal form (launched 2022) covers: home address, phone number, email, login credentials, financial info, government IDs, and medical info appearing in search results. This is distinct from the Remove Outdated Content tool.

### Severity & Automation

| Attribute | Value |
|-----------|-------|
| Severity | **High** |
| Remediation Effort | Medium — requires per-URL submissions |
| TraceBurn Automation | **Partial** — can auto-generate dork queries and discovered URLs; removal submission requires manual step (Google does not offer a programmatic removal API for non-Search-Console owners) |

## 2. Social Media Exposure

### Overview
Social platforms expose varying degrees of personal information by default. Even "private" accounts leak metadata through photos, location services, and third-party integrations.

### Default Exposure by Platform

#### LinkedIn
- Full name, employer, job title, and location are **public by default**.
- Profile photo is public.
- Connections list may expose your network.
- "Open to Work" banner is visible to all users if set to public.
- Contact info (email, phone) is visible to 1st-degree connections.
- **Metadata risk:** Profile activity timestamps can reveal work hours / timezone.

#### Facebook
- Name, profile photo, and cover photo are **always public** (cannot be made private).
- Posts, check-ins, and tagged photos default to Friends but can be set more openly.
- "About" section (hometown, employer, relationship status) defaults to Friends.
- Phone number and email are often added but hidden — however, Facebook uses them for ad targeting and people-search features.
- **Check-ins and location history** are visible to friends by default.

#### Twitter / X
- All tweets are **public by default** unless the account is set to "Protected."
- Display name, bio, and location field are always public.
- Location can be attached to individual tweets (opt-in but often accidentally enabled).
- **EXIF data:** Images uploaded before 2012 retained EXIF; X now strips EXIF on upload, but old images may still carry it.

#### Instagram
- Accounts are public by default.
- Tagged location on posts is public.
- Bio, username, and profile photo always public.
- Stories with location stickers are public on public accounts.
- **EXIF data:** Instagram strips EXIF on upload, but original files shared via DM or third-party exports retain it.

### Metadata Leaks

#### EXIF Data in Photos
EXIF (Exchangeable Image File Format) headers embed:
- GPS coordinates (latitude/longitude) — reveals home, work, or frequent locations
- Camera make/model and serial number
- Date/time of capture
- Software used (can reveal device type)

**Platforms that strip EXIF on upload:** Instagram, Twitter/X, Facebook (since 2012), WhatsApp
**Platforms that do NOT strip EXIF:** Flickr (preserves by default), some forums, direct file sharing, email attachments

**TraceBurn relevance:** If a user has shared photos on platforms that don't strip EXIF, their home address can be extracted. Tool: `exiftool` for local analysis.

#### Location Tags and Check-ins
- Facebook and Instagram check-ins create a public record of physical locations.
- Google Maps "Your Timeline" is private but Google uses it for ad targeting.
- Foursquare/Swarm check-ins were historically public — legacy data may still be indexed.

### Privacy Settings Checklist

#### LinkedIn
- [ ] Profile visibility → "Only you" or "Your connections" for contact info
- [ ] Turn off "Share profile updates with your network"
- [ ] Settings → Visibility → Profile viewing options → Anonymous
- [ ] Data privacy → Manage your data and activity → limit ad targeting
- [ ] Remove or hide email/phone from profile

#### Facebook
- [ ] Settings → Privacy → "Who can see your future posts?" → Friends
- [ ] Settings → Privacy → "Do you want search engines outside of Facebook to link to your profile?" → No
- [ ] Timeline and Tagging → review tags before they appear
- [ ] Profile → About → edit each section to Friends or Only Me
- [ ] Location History → Off (Facebook app settings)
- [ ] Remove phone number from profile (Settings → Mobile)

#### Twitter / X
- [ ] Settings → Privacy and Safety → "Protect your tweets" (if desired)
- [ ] Remove location from tweets: Settings → Privacy and Safety → Location information
- [ ] Settings → Privacy and Safety → Discoverability → uncheck email/phone search
- [ ] Review and delete old tweets with location data (use tweet archive + manual review)

#### Instagram
- [ ] Account → Private Account → On
- [ ] Settings → Privacy → Story → restrict who can reply
- [ ] Remove location from past posts (cannot bulk edit — must do per post)
- [ ] Settings → Privacy → Activity Status → Off

### Severity & Automation

| Attribute | Value |
|-----------|-------|
| Severity | **Medium–High** (varies by platform and settings) |
| Remediation Effort | Medium — settings changes are manual per platform |
| TraceBurn Automation | **Low** — can generate checklist and flag public profiles; cannot log into platforms to change settings (ToS restriction) |

## 3. Paste Sites

### Overview
Paste sites host arbitrary text content, often anonymously. They are a common dump location for leaked credentials, personal info, and breach data. Most offer no removal mechanism.

### Major Paste Sites

| Site | Notes |
|------|-------|
| Pastebin.com | Largest paste site. Public pastes indexed by Google. Removal only by original poster (if logged in) or via DMCA if content is copyrighted. |
| GitHub Gists (public) | Indexed by Google. Gists can be deleted by the creator. Cached versions may persist. |
| Ghostbin | Anonymous paste site. No account = no removal option. |
| Rentry.co | Markdown-based paste site. Edit/delete requires knowing the edit code set at creation. |
| Hastebin | Often self-hosted instances. Content is ephemeral but may be indexed if public. |
| JustPaste.it | Allows some removal — contact form available but inconsistent response. |
| ControlC.com | Large paste site, no removal mechanism. |
| Dpaste.com | Pastes expire by default (max 1 year), but can be indexed before expiry. |

### How to Search Paste Sites

```
# Google dorks for paste sites
site:pastebin.com "email@domain.com"
site:pastebin.com "Firstname Lastname"
site:pastebin.com "555-867-5309"
site:gist.github.com "email@domain.com"
site:ghostbin.com "Firstname Lastname"

# Broader search across all paste sites
"email@domain.com" site:pastebin.com OR site:gist.github.com OR site:ghostbin.com

# Searching for credential dumps
"email@domain.com" "password"
"email@domain.com" filetype:txt
```

**Note:** Google indexing of paste sites is inconsistent. A paste may exist but not be indexed. Dedicated paste-search tools (like Dehashed or IntelligenceX) index paste sites more comprehensively.

### What IS Possible for Removal

| Site | Removal Method |
|------|---------------|
| Pastebin.com | Delete if you created it (logged-in account); DMCA report form for copyrighted content; report form for private info (inconsistent) |
| GitHub Gists | Delete if you own the repo; DMCA takedown via GitHub's process if not |
| Ghostbin | No removal mechanism |
| Rentry.co | Delete with edit code (if you have it); no other mechanism |
| JustPaste.it | Contact form — report abuse |
| Most others | No removal — document and monitor only |

**Reality check:** For paste sites without removal mechanisms, the data is effectively permanent on that site. Mitigation is the only option: credential rotation, monitoring alerts, and noting the exposure in reports.

### Severity & Automation

| Attribute | Value |
|-----------|-------|
| Severity | **High** (often contains credentials or PII from breaches) |
| Remediation Effort | High — most sites offer no removal path |
| TraceBurn Automation | **Partial** — can automate Google dork searches and surface matches; removal is largely manual or impossible |

## 4. Breach Databases

### Overview
Breach databases aggregate data from known security breaches (credential dumps, leaked databases). Unlike other surfaces, breach data fundamentally **cannot be removed** — once leaked, copies proliferate across the dark web and private channels. The goal here is detection and mitigation, not removal.

### Have I Been Pwned (HIBP)

- **URL:** https://haveibeenpwned.com
- **Operator:** Troy Hunt (independent security researcher)
- **Coverage:** 13+ billion accounts across 800+ breaches (as of 2026)
- **Cost:** Free for individual email lookups; free API for personal use; paid API for commercial/bulk use

#### API Usage

```
# Check a single email address
GET https://haveibeenpwned.com/api/v3/breachedaccount/{email}
Headers:
  hibp-api-key: YOUR_API_KEY
  User-Agent: TraceBurn/1.0

# Get all breaches (no email required)
GET https://haveibeenpwned.com/api/v3/breaches

# Check if a password has been seen in breaches (k-Anonymity model)
GET https://api.pwnedpasswords.com/range/{first5hashchars}
# No API key required; privacy-preserving: only send first 5 chars of SHA-1 hash
```

**API Key:** Required for email lookups. Available at https://haveibeenpwned.com/API/Key
**Rate limiting:** 1 request per 1500ms per API key for email lookups
**Response:** Returns array of breach objects with name, date, data classes exposed, and description

#### Data Classes Tracked
HIBP tags each breach with the types of data exposed: email addresses, passwords, usernames, phone numbers, physical addresses, IP addresses, names, dates of birth, government IDs, financial info, etc.

### Dehashed

- **URL:** https://dehashed.com
- **Cost:** Paid — approximately $5.49/month (basic) to $179.99/month (enterprise) as of 2025
- **Coverage:** Broader than HIBP — includes leaked databases, paste sites, and some dark web sources
- **Search fields:** Email, username, name, phone, IP address, address, password hash

#### API Usage

```
# Search by email
GET https://api.dehashed.com/search?query=email%3Auser@domain.com
Headers:
  Authorization: Basic base64(email:api_key)
  Accept: application/json

# Search by name
GET https://api.dehashed.com/search?query=name%3A"Firstname+Lastname"

# Search by phone
GET https://api.dehashed.com/search?query=phone%3A5558675309
```

**Note:** Dehashed returns actual plaintext passwords and hashes where available — handle this data with extreme care. TraceBurn should never log or store raw password data from Dehashed results.

### IntelligenceX

- **URL:** https://intelx.io
- **Cost:** Free tier (limited); paid plans from ~$300/year for professional use
- **Coverage:** Dark web, paste sites, breach data, Tor, I2P, public data leaks, document repositories
- **Distinguishing feature:** Indexes historical data — finds information even after it's been removed from the source

#### What IntelligenceX Covers
- Breach databases and credential dumps
- Paste sites (Pastebin, GitHub Gists, and many others)
- Dark web sources (Tor, I2P hidden services)
- Leaked document repositories
- Historical web archives

#### API Usage

```
# Search (initiate)
POST https://2.intelx.io/intelligent/search
Body: {"term": "user@domain.com", "maxresults": 100, "media": 0, "sort": 2}

# Retrieve results (poll with returned ID)
GET https://2.intelx.io/intelligent/search/result?id={search_id}&limit=100
```

**API Key:** Required; available after account creation at intelx.io

### Firefox Monitor (Mozilla)

- **URL:** https://monitor.mozilla.org
- **Cost:** Free
- **Backend:** Powered by HIBP data
- **Value-add over raw HIBP:** Email digest alerts, dashboard UI, guidance on next steps
- **API:** No public API — uses HIBP's API under the hood
- **TraceBurn relevance:** Redundant with direct HIBP API integration; may be useful as a reference for user-facing UI patterns

### Mitigation Strategies (Breach Data Cannot Be Removed)

Since breach data proliferates and cannot be recalled, the appropriate response is:

1. **Password rotation:** Immediately change passwords for any service identified in a breach. Use a unique, randomly generated password per service.
2. **Credential manager adoption:** Recommend 1Password, Bitwarden, or similar. Document in TraceBurn output.
3. **Monitor for new breaches:** Set up HIBP notification emails for monitored addresses.
4. **Enable MFA:** Where a breached service supports it, enable TOTP or hardware key MFA.
5. **Check for password reuse:** If the breach exposed a password, assume it's being tested against other services (credential stuffing).
6. **Financial monitoring:** If financial data was in the breach, flag for credit monitoring.

### Severity & Automation

| Attribute | Value |
|-----------|-------|
| Severity | **High** |
| Remediation Effort | Low (detection) / High (full remediation across all affected services) |
| TraceBurn Automation | **High** — HIBP and Dehashed offer APIs; breach detection is fully automatable. Mitigation guidance can be generated programmatically based on data classes exposed. |
